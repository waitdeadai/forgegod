## Current Findings

### Resolved In This Audit Pass

- `forgegod loop` no longer auto-stages, commits, and pushes by default.
  Those behaviors are now explicitly gated behind
  `loop.auto_commit_success` and `loop.auto_push_success`.
- The benchmark runner no longer calls `Agent(..., project_dir=...)`.
  That constructor bug is fixed, and benchmark command parsing now handles
  quoted arguments safely.
- Worktree execution is now meaningfully scoped. Worker agents rebase
  `config.project_dir` to the assigned worktree, and runtime tool execution
  follows that scoped workspace instead of plain `Path.cwd()`.
- Filesystem tools are repo-scoped under agent execution. Escaping the active
  workspace root now returns an error, and configured `blocked_paths` are
  enforced.
- Missing `await` calls for episode persistence on max-turn and exception exits
  are fixed.
- Budget tracker write throughput regression is fixed. The rapid-write stress
  test that previously failed now passes.
- Benchmark auto-detection now covers the already-supported Gemini and
  DeepSeek providers, and it adds direct Moonshot / Kimi detection.
- ForgeGod now supports a direct `kimi` provider via Moonshot's official
  OpenAI-compatible API. This is present as an optional provider path, not a
  default model strategy.
- ForgeGod now supports a direct `zai` provider for Z.AI / GLM, with first-class
  support for the Coding Plan endpoint as well as the general API path.
- ForgeGod now supports native `openai-codex` routing through ChatGPT-backed
  Codex CLI login, exposed through `forgegod auth status`,
  `forgegod auth login openai-codex`, and `forgegod auth sync`.
- `forgegod init`, onboarding, and `forgegod auth sync` now write auth-aware
  model defaults. On existing local-only configs, `auth sync` can also lift the
  budget back to `normal` when cloud auth surfaces are present but Ollama is
  not.
- Those same surfaces now expose an explicit harness profile choice. Users can
  keep the recommended `adversarial` split-model path or choose `single-model`
  when they want every role pinned to one detected provider.
- ForgeGod now also has a true conversational root entrypoint. Running plain
  `forgegod` in an interactive terminal opens a natural-language task session,
  while `forgegod run` remains available for scripted/non-interactive use.
- That conversational root entrypoint now also bootstraps repo-local config on
  first use, so a user can install ForgeGod and start talking without having to
  run `forgegod init` first. The guided `init` flow still exists for users who
  want explicit onboarding and profile selection.
- That same conversational root entrypoint now honors the same runtime
  overrides as the scripted surfaces. `--terse`, model overrides,
  review toggles, permission modes, approval modes, and repeated allow-tool
  flags all survive the jump from `forgegod run` to plain `forgegod`.
- `codex exec` prompts now stream through stdin instead of Windows argv, which
  reduces compatibility risk when ForgeGod sends large tool schemas and prompts.
- ForgeGod now supports `DESIGN.md` as a first-class frontend control surface:
  `forgegod design` can import presets from `awesome-design-md`, and the agent
  loads repo-local `DESIGN.md` into the prompt for frontend tasks.
- ForgeGod now ships `forgegod contribute`, which reads `CONTRIBUTING.md`,
  AGENTS-style repo rules, and optionally GitHub `good first issue` /
  `help wanted` issues before planning or executing a contribution.
- Shell hardening is meaningfully stronger than the earlier baseline.
  Standard/strict modes now isolate process home/cache/config directories,
  keep execution workspace-scoped, block shell operators such as chaining,
  pipes, redirection, and command substitution, and strict mode uses
  allowlisted single-process exec rather than a shell parser.
- Strict mode now uses a real Docker sandbox backend instead of host execution.
  Commands run in a no-network, read-only-root container with the workspace
  bind-mounted, or they are blocked if the Docker daemon or sandbox image is
  unavailable.
- ForgeGod now exposes `forgegod permissions`, which surfaces the effective
  permission policy, blocked tools, and allowlist override from the CLI.
- ForgeGod now supports approval modes layered on top of permission modes:
  `deny`, `prompt`, and `approve`. The CLI prompt path is wired into
  `forgegod run`, and approved tool overrides are covered by deterministic
  tests instead of being left as a manual behavior.
- Prompt approval is now explicitly serialized across loop/worktree surfaces.
  `forgegod loop --approval-mode prompt` rejects `--workers > 1`, and
  `WorktreePool` now refuses prompt mode unless it runs with one worker and a
  supplied approval callback.
- `forgegod loop` now closes its `ModelRouter` cleanly on exit instead of
  leaving cleanup to process teardown.
- `forgegod loop --workers > 1` now really executes through isolated git
  worktrees instead of multiplexing agents in the same workspace. It now fails
  fast unless the repo has at least one commit, and new files created inside a
  worker worktree can be reviewed and patched back into the main workspace.
- The standalone `git_worktree_create` tool now follows that same behavior.
  ForgeGod no longer has one worktree flow in `loop/worktree` and a stale,
  weaker branch-plus-HEAD flow in the git tool surface.
- OpenAI direct routing now honors a configurable `[openai].base_url`, which
  enables deterministic local OpenAI-compatible harnesses instead of forcing
  live-vendor calls for end-to-end CLI verification.
- Loop execution now has deterministic parity coverage too. The CLI mock
  harness exercises successful and permission-blocked `forgegod loop` runs
  against a local OpenAI-compatible provider instead of stopping at `run`.
- That same CLI mock parity harness now also exercises isolated parallel
  worktree execution through `forgegod loop --workers 2`, proving that the
  product path uses a real worktree, applies the resulting patch back into the
  main workspace, and cleans up worktree directories afterward.
- ForgeGod now has explicit memory switches for harness isolation:
  `memory.enabled` and `memory.extraction_enabled`. That prevents hidden
  post-task memory-model calls from polluting deterministic loop parity runs.
- ForgeGod now also has deterministic CLI coverage for auth-aware provider
  selection in `forgegod auth sync`, including cloud-budget normalization and
  the warning path for Codex-as-coder defaults.
- ForgeGod doctor now checks strict sandbox readiness directly and points users
  to a non-technical setup guide so Docker/image failures are less opaque.
- Generated-code validation is no longer purely advisory. In `strict` mode,
  suspicious writes and edits are blocked before they land on disk.
- Repeated validation and memory-write hot paths were optimized without
  relaxing policy, restoring the full stress suite to green status.
- ForgeGod now has a true CLI mock parity harness on top of the internal
  scripted parity layer. `forgegod run` is verified against a deterministic
  local OpenAI-compatible endpoint, and permission-denied write tasks now fail
  fast with an actionable error instead of burning all remaining turns.
- That CLI mock parity harness now covers strict sandbox semantics too:
  allowed strict-mode bash execution must route through the real sandbox
  interface, and missing strict backends must surface back to the CLI as a
  deterministic failure instead of looping until max turns.
- ForgeGod now also has an opt-in real Docker strict integration test. With
  `FORGEGOD_RUN_DOCKER_STRICT_TESTS=1` and Docker ready, the harness proves a
  real strict backend roundtrip without monkeypatching.
- ForgeGod now has a first-class deterministic harness eval surface:
  `forgegod evals`. It is distinct from `forgegod benchmark`, ships built-in
  CLI behavior cases, saves per-case traces, and gives the repo a reusable way
  to grade natural-language chat, terse mode, approval flows, permission
  denials, completion-gate behavior, loop/worktree behavior, and strict-sandbox
  interface handling.
- Strict Node/web bootstrap now isolates `node_modules` into a named Docker
  volume, including on bootstrap commands such as `create-next-app`, instead of
  relying on host bind-mounted dependency trees.
- The strict dependency readiness check now treats the manifest hash plus the
  Docker volume as the real source of truth, instead of trusting host-side
  `node_modules` leftovers after resets or volume cleanup.
- Agent execution now injects bounded repo context docs (`docs/README.md`,
  `docs/PRD.md`, `docs/STORIES.md`, `docs/ARCHITECTURE.md`, `docs/RUNBOOK.md`)
  so task execution can follow the same checked-in system of record as planning.
- ForgeGod now degrades cleanly when optional HTTP/2 support is absent in the
  host environment. If `httpx` is installed without the `h2` extra, the router
  logs once and falls back to HTTP/1.1 instead of failing provider calls.
- CI should be green again on the verified local baseline, including a full
  rerun on Python 3.11: tests, lint, and build all pass.
- ForgeGod CLI now has a shared branded UX layer instead of separate utilitarian
  consoles. Onboarding, doctor, and TUI surfaces now share ForgeGod colors, and
  `forgegod run` / `forgegod loop` narrate work in plain language instead of
  surfacing successful transport chatter as the main user experience.

### Remaining Risks

- Standard mode is still a host-local guarded workflow, not a true sandbox.
- Strict mode now has real container isolation and no-network execution, but
  ForgeGod does not yet provide microVM isolation, custom seccomp profiles, or
  a stronger backend than the local Docker Engine.
- Generated-code validation is blocking in `strict` mode and advisory in
  `standard` mode. That is a clear improvement, but it still leaves normal
  operation less restrictive than a hardened sandbox profile.
- `BENCHMARKS.md` remains a historical snapshot. The benchmark runner path is
  fixed, but the published benchmark claims have not yet been regenerated from
  the repaired code path.
- Live benchmarks still depend on configured local/cloud models. The benchmark
  command now discovers more providers, but in an empty environment it still
  exits with no runnable models.
- OpenAI Codex subscription is verified for planner/reviewer/adversary flows,
  but coder-loop use is still experimental and should be benchmarked before it
  becomes the default remote coding surface.
- ForgeGod now also supports an explicit OpenAI-first adversarial profile.
  `preferred_provider = "openai"` lets the harness bias planner/researcher/
  coder roles toward GPT-5.4 API models while still using Codex subscription
  auth as the reviewer/adversary surface.
- Direct library calls to tool functions in tests or custom scripts remain less
  constrained than agent-driven runtime, because workspace scoping is applied
  via agent tool context.
- A local strict-sandbox smoke still depends on Docker Desktop actually exposing
  a live daemon. In the latest verification session, ForgeGod correctly
  classified the blocker as "daemon not ready" rather than "CLI missing", but
  the real Docker smoke could not run until the engine pipe comes back.

## Architecture Delta Vs. claw-code

Research on `2026-04-08` against `https://github.com/ultraworkers/claw-code`
and its `USAGE.md`, `PARITY.md`, and mock parity harness shows a split result:

- ForgeGod is already ahead on provider breadth, native auth surfaces, role
  routing, memory, and PRD-driven autonomous looping.
- ForgeGod has materially narrowed the deterministic harness gap. It now has
  both an internal scripted parity harness and an end-to-end CLI mock parity
  harness with explicit permission-denial, tool-roundtrip, completion-gate,
  and real `loop` scenarios.
- ForgeGod has also narrowed the permission ergonomics gap with
  `forgegod permissions`, static CLI permission flags, and a first-class
  serial interactive `approve/deny` flow. The remaining gap is broader
  deterministic coverage of those approvals in `loop` and any future
  first-class worktree execution path.
- The most important reliability gap remains task completion semantics. A
  recent Tarot benchmark showed that ForgeGod could report completion without
  satisfying visible acceptance criteria. That is a harness defect, not a
  product defect.

## Local Work In Progress (2026-04-08)

The current local workstream has started addressing that completion-semantics
gap:

- `forgegod.agent` now tracks post-edit verification evidence and final diff
  review before accepting completion on implementation tasks.
- `forgegod run --review` no longer treats the review flag as cosmetic; local
  changes now block completion when the reviewer returns `revise` or `reject`.
- `forgegod.loop` no longer relies on sample-only review when a story has
  acceptance criteria; those stories now force review by default.
- `forgegod.loop` and `forgegod.worktree` now agree on prompt approval
  semantics: serialized prompt approval is allowed, but parallel prompt
  approval is rejected as an invalid harness state instead of being left
  ambiguous.
- `forgegod.loop` can now disable memory and LLM-backed memory extraction
  explicitly, which gives parity harnesses and future benchmarks a clean way
  to isolate loop behavior without hidden extra provider calls.
- `forgegod auth sync` is now covered as a deterministic product surface,
  not just a helper chain. The harness verifies that detected auth surfaces
  rewrite role defaults and normalize cloud budget settings correctly.
- `forgegod.agent` now classifies observational shell/report tasks as
  read-only work instead of defaulting them into implementation completion
  gates. That fixes a real harness defect exposed by strict-sandbox parity:
  tasks like "Run python --version and report it" no longer loop forever
  waiting for file changes that should never happen.
- `forgegod.worktree` now preserves raw `git diff --binary` output instead of
  trimming it through a text helper, which fixes corrupt patch application for
  newly created files when patching worktree results back into the main repo.
- `forgegod.worktree.ensure_parallel_ready()` now explicitly requires at least
  one git commit before parallel workers can start, so users get a precise
  error instead of burning loop iterations on `fatal: invalid reference: HEAD`.
- `forgegod.tools.git.git_worktree_create()` now enforces that same preflight
  and uses `git worktree add -b ...` directly, with dedicated tests for both
  the no-commit failure and the successful roundtrip path.

## Recommended Next Work

1. Add an opt-in real-Docker tier to `forgegod evals` so strict backend
   coverage can graduate from a separate smoke into a release-gated eval
   layer.
2. Add a stronger strict backend such as Docker Sandboxes or another
   microVM/syscall-confined runtime if ForgeGod is intended to operate on
   untrusted repos.
3. Expand the opt-in real Docker strict integration coverage beyond the
   current happy-path bash roundtrip, so backend parity includes more than one
   safe command shape.
4. Regenerate benchmark claims or remove leaderboard-style implied currency from
   public docs until fresh runs exist.
5. Add direct tests for loop auto-commit flags and for any future worktree
   merge path so the repaired behavior stays locked in.
6. Decide whether OpenAI Codex coder-loop behavior should stay experimental or
   graduate after broader repo-local evals plus benchmarks.

## Residual Risk Summary

ForgeGod is in a materially stronger state than the earlier 2026-04-07
baseline. The repo now has a green test suite, green lint, green build,
safer loop defaults, scoped tool execution, repaired benchmark plumbing,
a real strict Docker sandbox path, and a green stress suite. The main remaining
gap is no longer correctness drift; it is hardening depth: benchmark freshness,
stronger-than-container isolation, and how restrictive `standard` mode should become.
