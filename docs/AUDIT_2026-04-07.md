## Current Findings

### Resolved In This Audit Pass

- `forgegod loop` no longer auto-stages, commits, and pushes by default.
  Those behaviors are now explicitly gated behind
  `loop.auto_commit_success` and `loop.auto_push_success`.
- The benchmark runner no longer calls `Agent(..., project_dir=...)`.
  That constructor bug is fixed, and benchmark command parsing now handles
  quoted arguments safely.
- Worktree execution is now meaningfully scoped. Worker agents rebase
  `config.project_dir` to the assigned worktree, and runtime tool execution
  follows that scoped workspace instead of plain `Path.cwd()`.
- Filesystem tools are repo-scoped under agent execution. Escaping the active
  workspace root now returns an error, and configured `blocked_paths` are
  enforced.
- Missing `await` calls for episode persistence on max-turn and exception exits
  are fixed.
- Budget tracker write throughput regression is fixed. The rapid-write stress
  test that previously failed now passes.
- Benchmark auto-detection now covers the already-supported Gemini and
  DeepSeek providers, and it adds direct Moonshot / Kimi detection.
- ForgeGod now supports a direct `kimi` provider via Moonshot's official
  OpenAI-compatible API. This is present as an optional provider path, not a
  default model strategy.
- ForgeGod now supports a direct `zai` provider for Z.AI / GLM, with first-class
  support for the Coding Plan endpoint as well as the general API path.
- ForgeGod now supports native `openai-codex` routing through ChatGPT-backed
  Codex CLI login, exposed through `forgegod auth status`,
  `forgegod auth login openai-codex`, and `forgegod auth sync`.
- `forgegod init`, onboarding, and `forgegod auth sync` now write auth-aware
  model defaults. On existing local-only configs, `auth sync` can also lift the
  budget back to `normal` when cloud auth surfaces are present but Ollama is
  not.
- `codex exec` prompts now stream through stdin instead of Windows argv, which
  reduces compatibility risk when ForgeGod sends large tool schemas and prompts.
- ForgeGod now supports `DESIGN.md` as a first-class frontend control surface:
  `forgegod design` can import presets from `awesome-design-md`, and the agent
  loads repo-local `DESIGN.md` into the prompt for frontend tasks.
- ForgeGod now ships `forgegod contribute`, which reads `CONTRIBUTING.md`,
  AGENTS-style repo rules, and optionally GitHub `good first issue` /
  `help wanted` issues before planning or executing a contribution.
- Shell hardening is meaningfully stronger than the earlier baseline.
  Standard/strict modes now isolate process home/cache/config directories,
  keep execution workspace-scoped, block shell operators such as chaining,
  pipes, redirection, and command substitution, and strict mode uses
  allowlisted single-process exec rather than a shell parser.
- Strict mode now uses a real Docker sandbox backend instead of host execution.
  Commands run in a no-network, read-only-root container with the workspace
  bind-mounted, or they are blocked if the Docker daemon or sandbox image is
  unavailable.
- Generated-code validation is no longer purely advisory. In `strict` mode,
  suspicious writes and edits are blocked before they land on disk.
- Repeated validation and memory-write hot paths were optimized without
  relaxing policy, restoring the full stress suite to green status.
- CI should be green again on the verified local baseline: tests, lint, and
  build all pass.

### Remaining Risks

- Standard mode is still a host-local guarded workflow, not a true sandbox.
- Strict mode now has real container isolation and no-network execution, but
  ForgeGod does not yet provide microVM isolation, custom seccomp profiles, or
  a stronger backend than the local Docker Engine.
- Generated-code validation is blocking in `strict` mode and advisory in
  `standard` mode. That is a clear improvement, but it still leaves normal
  operation less restrictive than a hardened sandbox profile.
- `BENCHMARKS.md` remains a historical snapshot. The benchmark runner path is
  fixed, but the published benchmark claims have not yet been regenerated from
  the repaired code path.
- Live benchmarks still depend on configured local/cloud models. The benchmark
  command now discovers more providers, but in an empty environment it still
  exits with no runnable models.
- OpenAI Codex subscription is verified for planner/reviewer/adversary flows,
  but coder-loop use is still experimental and should be benchmarked before it
  becomes the default remote coding surface.
- Direct library calls to tool functions in tests or custom scripts remain less
  constrained than agent-driven runtime, because workspace scoping is applied
  via agent tool context.

## Documentation Drift

The earlier audit found several visible mismatches between docs and runtime.
Current status:

- README and README.es now describe the repaired baseline:
  `23` tools, `8` provider families, `9` route surfaces, `449` collected tests,
  green lint, green build,
  and no default auto-push in loop mode.
- `BENCHMARKS.md` should stay clearly labeled as historical until fresh results
  are regenerated from the repaired benchmark pipeline.
- `docs/index.html` is now aligned with the current provider count and audited
  test baseline, but operational docs should still treat the audit and repo
  docs as the source of truth for runtime behavior. Repo policy now requires
  updating the site whenever public-facing claims change.

## Recommended Next Work

1. Add a stronger strict backend such as Docker Sandboxes or another
   microVM/syscall-confined runtime if ForgeGod is intended to operate on
   untrusted repos.
2. Add doctor/onboarding checks for Docker daemon readiness and required
   sandbox images so strict mode fails less opaquely.
3. Regenerate benchmark claims or remove leaderboard-style implied currency from
   public docs until fresh runs exist.
4. Add direct tests for worktree isolation and loop auto-commit flags so the
   repaired behavior stays locked in.
5. Decide whether `standard` mode should block a larger subset of suspicious
   generated-code writes instead of only warning.
6. Decide whether OpenAI Codex coder-loop behavior should stay experimental or
   graduate after broader repo-local benchmarks.

## Residual Risk Summary

ForgeGod is in a materially stronger state than the earlier 2026-04-07
baseline. The repo now has a green test suite, green lint, green build,
safer loop defaults, scoped tool execution, repaired benchmark plumbing,
a real strict Docker sandbox path, and a green stress suite. The main remaining
gap is no longer correctness drift; it is hardening depth: benchmark freshness,
stronger-than-container isolation, and how restrictive `standard` mode should become.
